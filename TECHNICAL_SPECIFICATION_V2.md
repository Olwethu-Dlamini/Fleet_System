# 🚛 Vehicle Scheduling System: Exhaustive Technical Specification (V2)

This document provides a comprehensive, low-level technical analysis of the Vehicle Scheduling System. It is designed for developers, architects, and system maintainers who require a deep understanding of the platform's inner workings, from raw SQL queries to high-level state management patterns.

---

## 1. Architectural Foundations

The system is built on a **Modern Decoupled Full-Stack Architecture**, optimized for high availability, transactional integrity, and cross-platform accessibility.

### 1.1 Tech Stack Deep Dive
*   **Backend Runtime**: Node.js (v20+) with the Express framework.
*   **Database Engine**: MySQL 8.0 (Community Edition), chosen for its robust support for ENUM types and complex JOIN performance.
*   **Frontend Framework**: Flutter (v3.19+), targeting Android, iOS, and Web.
*   **Communication Protocol**: RESTful JSON API over HTTPS.
*   **Authentication Strategy**: Stateless JWT (JSON Web Tokens) with HS256 signing.

### 1.2 Deployment Strategy
The system is containerized using **Docker** and deployed on **AWS EC2**.
*   **Host Networking**: The backend container uses `--network=host` to bypass Docker's virtual bridge. This allows direct, low-latency access to the MySQL service running on the EC2 host's `localhost`.
*   **CORS Policy**: The backend implements a dynamic origin-validation function. It specifically allows all `localhost` and `127.0.0.1` origins on any port, ensuring developers can debug the Flutter Web target without proxy workarounds.

## 2. Database Schema & Data Integrity

The database (`vehicle_scheduling`) is normalized to the 3rd Normal Form (3NF) to minimize redundancy and prevent update anomalies.

### 2.1 Core Entities

#### 2.1.1 The `jobs` Table
The central registry for all work orders.
- `id` (INT UNSIGNED, PK): Unique internal ID.
- `job_number` (VARCHAR): Format `JOB-YYYY-XXXX`. Generated by the backend using a specialized sequence generator.
- `job_type` (ENUM): `installation`, `delivery`, `miscellaneous`.
- `current_status` (ENUM): `pending`, `assigned`, `in_progress`, `completed`, `cancelled`.
- `priority` (ENUM): `low`, `normal`, `high`, `urgent`.

#### 2.1.2 The `job_assignments` Table
A junction table resolving the many-to-one relationship between jobs, vehicles, and users.
- `job_id`: FK to `jobs`.
- `vehicle_id`: FK to `vehicles`.
- `driver_id`: FK to `users` (technician role).
- `notes`: TEXT field for dispatcher instructions.

### 2.2 Relational Logic
The schema uses **Trigger-like constraints** via Application Logic:
1.  **Status Propagation**: When an assignment is created in `job_assignments`, the `jobs.current_status` is atomically updated to `assigned` within a database transaction.
2.  **Referential Cleanup**: `ON DELETE CASCADE` is applied to assignments, ensuring that if a job is deleted, its historical assignment record is also purged.

## 3. Backend Logic & Controller Deep Dive

### 3.1 Authentication & Permissions (`authController.js`)
The authentication system is more than a simple password check; it is a **Permission Computation Engine**.

1.  **Role Normalization**: Before generating a JWT, the server runs `_normaliseRole()`. This maps legacy database strings (like `driver`) to canonical application roles (`technician`).
2.  **Dynamic Permission Injection**: The server does NOT store permissions in the database. Instead, it maintains a `PERMISSIONS` map in `constants.js`. At login, the server performs a reverse lookup:
    ```javascript
    const userPermissions = Object.entries(PERMISSIONS)
      .filter(([, roles]) => roles.includes(userRole))
      .map(([perm]) => perm);
    ```
    This allows developers to add new features and gate them by role by simply updating a single constant file, without needing a database migration.

### 3.2 Assignment Engine (`jobAssignmentController.js`)
This controller manages the most complex logic in the system: **Conflict Detection**.

*   **Atomic Multi-Assignment**: The `assignTechnicians` function allows assigning multiple drivers to a single job.
*   **Admin Force-Override**: A critical feature implemented for senior staff. If a driver is already assigned to another job during the same time window, a standard dispatcher request will fail with a `409 Conflict`. However, if an Admin sends `force_override: true`, the backend will:
    1.  Identify the conflicting assignment.
    2.  Delete the conflicting row from `job_assignments`.
    3.  Create the new assignment.
    4.  Update both jobs' statuses accordingly.

## 4. Frontend Architecture & State Management

### 4.1 State Management via Provider
The app uses `MultiProvider` at the root (`main.dart`) to manage three core slices of state:
1.  **`AuthProvider`**: Manages the user's session, token storage (via `shared_preferences`), and role-based UI gating.
2.  **`JobProvider`**: Handles the massive list of jobs. It implements **Optimistic UI Updates**—when a technician changes a job status, the local list is updated *before* the API call completes, making the app feel instantaneous.
3.  **`VehicleProvider`**: Manages fleet data and availability.

### 4.2 Robust Networking: The `ApiService` Singleton
Communicating with a mobile backend requires resilience against network jitter.
- **Resilient Decoding**: The `_handleResponse` method prevents the common "TypeError: _InternalLinkedHashMap<String, dynamic> is not a subtype of type..." error. If the backend returns a JSON list or a raw string, the service wraps it in a standard `Map` structure so the rest of the app doesn't crash.
- **Header Injection**: The `ApiService` uses a private `_headers` getter that automatically pulls the latest Bearer token from the `AuthProvider`, ensuring developers never have to manually pass tokens to individual API calls.

### 4.3 Data Modeling (`lib/models/job.dart`)
The `Job` model is a "Smart Model." It includes:
- **`fromJson` with Type Safety**: Uses internal `_parseInt` and `_parseDate` helpers to gracefully handle cases where the MySQL driver might return a string instead of an integer for an ID.
- **Virtual Properties**: Getters like `formattedTimeRange` (converting `09:00:00` to `09:00`) ensure that the UI code remains clean and focused only on layout.

## 5. Security & Data Protection

### 5.1 Transport Security
All communication is designed for HTTPS. The `AppConfig.baseUrl` is the single source of truth for all API calls, making it easy to switch from staging to production.

### 5.2 Input Validation
The system implements **Defense in Depth**:
1.  **Flutter Validation**: Forms use `TextFormField.validator` to catch empty fields or malformed emails before the network is even touched.
2.  **API Validation**: Controllers in the backend use `isNaN()` and presence checks to verify every incoming parameter.
3.  **Database Validation**: ENUM columns and `NOT NULL` constraints provide the final layer of protection against data corruption.

## 6. Key Code Implementation Examples

### 6.1 The "Safe Parser" Pattern (Flutter)
Found in every model, this prevents crashes when dealing with null or mismatched types from the database.
```dart
static int _parseInt(dynamic v) {
  if (v == null) return 0;
  if (v is int) return v;
  return int.tryParse(v.toString()) ?? 0;
}
```

SchedulerBinding.instance.addPostFrameCallback((_) {

## 7. Operational Workflow: The Life of a Job

1.  **Ingestion**: `POST /api/jobs` creates a `pending` record.
2.  **Matching**: Dispatcher calls `GET /api/vehicles` and `GET /api/users?role=technician`.
3.  **Binding**: `POST /api/job-assignments/assign` binds the job, vehicle, and technician. The job status flips to `assigned`.
4.  **Activation**: Technician sees the job in "My Jobs." Clicking "Start" calls `PUT /api/job-status/update`, flipping the status to `in_progress`.
5.  **Finalization**: `PUT /api/job-status/update` with status `completed` archives the assignment and marks the vehicle as available for the next slot.

## 8. Troubleshooting Reference

| Error | Likely Root Cause | Investigation Step |
| :--- | :--- | :--- |
| **TypeError in Flutter** | Backend returned a JSON array instead of an Object. | Check `ApiService._handleResponse` logs. |
| **401 Unauthorized** | JWT has expired (8h) or `injectToken()` wasn't called. | Verify `AuthProvider.isLoggedIn`. |
| **409 Conflict** | Double-booking attempt on a vehicle/driver. | Check `job_assignments` for overlapping TIME windows. |
| **White Screen on Login** | `AuthGate` failed to resolve `checkAuthStatus()`. | Check `shared_preferences` storage. |
